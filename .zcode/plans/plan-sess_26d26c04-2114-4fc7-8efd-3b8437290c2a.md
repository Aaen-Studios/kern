# Fix two regressions from the last pass, then continue

## Bug 1 — Settings view crashes: `model.getLanguageId is not a function`

**Root cause:** `DiffViewer.tsx` creates two standalone Monaco models via `editor.createModel(...)` then, on mount, casts the React `<DiffEditor>`'s editor handle to `IStandaloneDiffEditor` and calls `diffEd.setModel({ original, modified })`. On this Monaco/`@monaco-editor/react` version that manual model management breaks (the mounted editor's model is replaced incorrectly → `getLanguageId` is called on a half-initialized model). The crash is triggered simply by the `DiffViewer` module being imported (it runs `monacoEditor.editor.defineTheme` + model setup at module scope), which is why it breaks the **whole view**, not just the diff tab.

**Fix:** Rewrite `DiffViewer.tsx` to use the supported, declarative API — pass `original` and `modified` strings **directly as props** to `<DiffEditor>`, and drop the manual `createModel`/`setModel`/cast entirely. `@monaco-editor/react`'s `DiffEditor` accepts `original`/`modified`/`language` props and manages models itself. Keep the theme registration but guard it so re-import is a no-op. Net effect: ~40 lines simpler, no module-scope model creation, no crash.

## Bug 2 — Ad-hoc terminal commands return nothing

**Root cause (backend):** `run_terminal_command` spawns `Command::new(&command)` directly. On Windows, `dir`, `echo`, `type`, `set` etc. are `cmd.exe` **builtins**, not executables — so `spawn()` fails with "program not found" immediately. Same for `ls`/`echo` aliases via `sh` on Unix in some setups.

**Root cause (frontend):** `ServerDetailView.tsx:205` calls it with `void invoke(...)` — fire-and-forget — so the spawn failure is swallowed and the user sees nothing (no error, no output).

**Fix (backend):** Add a `pub(crate) build_adhoc_shell_command(raw_line)` helper in `process.rs` that runs the **verbatim** user input through the OS shell (`cmd.exe /C "<line>"` on Windows, `sh -c "<line>"` on Unix) — so builtins, pipes, and redirects all work. This is distinct from the existing `build_shell_command` (private, used for lifecycle launchers, which rewrites bare names to `.bat`/`.sh` — wrong for ad-hoc input). Rewrite `run_terminal_command` to take a single `line: String` (the full trimmed input) and spawn via this helper; keep the instance lookup, `.env` layering, window suppression, stdout/stderr streaming, and `latest.log` append exactly as now.

**Fix (frontend):** Change `ServerDetailView.tsx` to (a) pass the whole trimmed line as `line` instead of splitting into `command`/`args`, and (b) replace `void invoke(...)` with `await` + a `.catch` that surfaces the failure as a `[error] ...` line in the terminal via `pushLine`, so a bad command tells the user instead of vanishing.
